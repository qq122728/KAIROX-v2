# KYC 上传修复归档

状态：**Closed**

## 问题原因

生产 KYC 提交失败的根因是 Nginx 默认请求体限制约 1 MB。两张压缩图片组成的 multipart 请求体约为 1.1–1.8 MB，因此请求在 Nginx 层直接返回 `413 Request Entity Too Large`，未到达 Next.js 或 SQLite。

## 修复方案

- KYC 前端和后端统一单文件上限为 2 MB。
- `compressImage` 使用最大 1920px 和有限 JPEG 质量策略：`0.78 → 0.62 → 0.48`。
- 压缩后仍超限时，前端阻止请求并显示明确提示。
- `/api/kyc` 对 Front/Back 单文件超限返回 JSON `413`。
- Nginx 主域名 `/api/kyc` 设置 `client_max_body_size 5m`。
- 保留同名文件独立处理逻辑，不使用文件名判重。

## 验证结果

- 同名 JPEG + JPEG：提交成功，Front/Back 数据独立。
- 同名 PNG + PNG：提交成功，Front/Back 数据独立。
- 单文件超过 2 MB：后端返回 JSON 413。
- 总请求超过 5 MB：Nginx 返回 413。
- 未登录：401。
- `npm run lint`：通过。
- `npm run build`：通过。
- 数据库 `integrity_check`：ok。
- `foreign_key_check`：无错误。
- 生产首页/Login/Register：200。
- JS/CSS 缺失：0。

## 后续维护建议

1. 新图片入口优先复用 `compressImage`，不要复制压缩逻辑。
2. API 单文件限制必须不高于前端实际可提交上限。
3. Nginx 备份文件不得放入 `sites-enabled`，避免被当作活动配置加载。
4. 每次 KYC 发布继续验证单文件 413、总请求 413 和同名文件提交。
5. 真机回归重点覆盖 iPhone Safari 的 HEIC 拍照路径。
